Module 1 · Compute

Amazon EC2 & Compute Services

EC2 is AWS's virtual machine service — flexible, scalable, and foundational. Learn instance types, pricing models, and how Auto Scaling + Load Balancing keep apps resilient.

What is Amazon EC2?

Amazon Elastic Compute Cloud (EC2) lets you rent virtual servers in the cloud. Unlike physical servers you buy upfront, EC2 gives you on-demand compute capacity — spin up instances in minutes, pay only for what you use, and shut them down when done. AWS handles the physical hardware; you control the OS, software, and configuration.
☕ Coffee Shop Analogy
EC2 is like ordering espresso — you request it when you need it (on-demand), choose the size (instance type), and pay per cup. No need to own the machine or the building.

EC2 Instance Types — Choose the Right Hardware

Different workloads need different hardware. AWS offers 5 families. Think of each as a specialized tool:

General Purpose
Balanced CPU, memory, networking. The "Swiss Army knife" for web servers, small databases, and dev environments.
e.g., t3, m5
Compute Optimized
High-performance CPUs for batch processing, gaming servers, scientific modeling, and high-performance web servers.
e.g., c5, c6g
Memory Optimized
Large RAM for in-memory databases, real-time analytics, and big data processing where data lives in memory.
e.g., r5, x1
Accelerated Computing
GPU/FPGA hardware for machine learning, graphics rendering, and floating-point calculations that can't be done efficiently on CPUs.
e.g., p3, g4
Storage Optimized
High disk I/O for data warehouses, distributed file systems, and OLTP systems that need fast sequential read/write.
e.g., i3, d2

EC2 Pricing Models — Pay for What You Need

ModelBest ForSavings
On-DemandUnpredictable workloads, testing, getting started. No commitment, highest flexibility.Baseline price
Savings PlansConsistent workloads. Commit to $/hr for 1 or 3 years on any instance family.Up to 66%
Reserved InstancesSteady-state workloads. Reserve specific instance type/region for 1 or 3 years.Up to 75%
Spot InstancesFlexible, fault-tolerant jobs — batch processing, data analysis. AWS can reclaim with 2-min notice.Up to 90%
Dedicated HostsCompliance needs. Physical server dedicated to you — no shared hardware with other customers.0% (most expensive)
Memory trick: On-Demand → anytime, no strings. Savings Plans → commit to spend. Reserved → commit to specific instance. Spot → cheapest but interruptible. Dedicated → your own physical machine.

Auto Scaling — Never Over or Under Provision

EC2 Auto Scaling automatically adds or removes EC2 instances based on demand. If traffic spikes (Black Friday), more instances launch. When traffic drops, unneeded instances terminate — you stop paying for idle capacity.
Dynamic Scaling
Responds to real-time demand changes. Scale out when CPU > 70%, scale in when CPU < 30%.
Predictive Scaling
Uses ML to forecast future demand and provisions capacity ahead of time. Great for known patterns like business hours.
Min / Max / Desired
You set the floor (minimum instances always running), ceiling (maximum allowed), and starting point (desired capacity).

Elastic Load Balancing (ELB)

ELB sits in front of your EC2 fleet and distributes incoming traffic across multiple instances. If one instance fails, ELB stops sending it traffic. When Auto Scaling adds new instances, ELB automatically starts routing to them. Together, ELB + Auto Scaling = high availability at scale.

Module 2 · Messaging & Decoupling

SQS, SNS & EventBridge

Tightly coupled apps fail together. Loosely coupled apps survive partial failures. Learn how AWS messaging services let components communicate without direct dependencies.

Monolithic vs Microservices Architecture

ArchitectureHow it worksProblem
MonolithicAll components tightly bundled — database, UI, business logic in one app. A failure in one part can crash everything.Brittle, hard to scale independently
MicroservicesComponents are loosely coupled and communicate via messages/APIs. One component failing doesn't bring down others.More complex to build initially

The Three Messaging Services — Quick Comparison

Amazon SQS — Queue
Messages sit in a queue until a consumer picks them up. The sender doesn't need the receiver to be online. Perfect when the receiver is sometimes busy or unavailable. One consumer per message.
1-to-1 delivery
Amazon SNS — Topics
Publisher sends one message to a "topic" and all subscribers (email, Lambda, SQS, SMS, HTTP) receive it simultaneously. Fan-out distribution.
1-to-many delivery
Amazon EventBridge
Routes events from AWS services, custom apps, and SaaS tools to targets. More powerful than SNS with filtering and transformation. Event-driven architecture hub.
Event routing
When to use what:
• SQS → "Process this task when you're free" (order queue, job queue)
• SNS → "Tell everyone about this now" (notifications, alerts)
• EventBridge → "Route this event to the right service based on rules" (complex workflows)

SQS Deep Dive — How Queuing Prevents Bottlenecks

Imagine a customer support system: the support agent adds tickets to a queue, the technical specialist picks them up when ready. Even if the specialist is busy, the agent keeps adding tickets. The queue acts as a buffer — no messages lost, no blocking.

Module 3 · Serverless & Containers

Lambda, ECS, EKS & More

The progression from managing everything → managed services → serverless. As you move right, you focus more on code, less on infrastructure.

Three Levels of Compute Management

TypeExampleYou manageAWS manages
UnmanagedAmazon EC2OS, patches, scaling, configPhysical hardware
ManagedELB, SNS, SQSConfig & deploymentInfrastructure, scaling, HA
Fully Managed (Serverless)AWS LambdaJust your codeEverything else
Coffee Analogy: Unmanaged = custom espresso machine (full control, full maintenance). Managed = pod coffee maker (configure it, press button). Serverless = order from a barista (just describe what you want, they handle everything).

AWS Lambda — Serverless Functions

Lambda lets you run code in response to events without managing any servers. You write the code, define a trigger, and Lambda handles scaling, availability, and execution automatically. You pay only for the milliseconds your code runs.
How Lambda Works
1. Upload your code as a Lambda function
2. Set a trigger (S3 upload, API call, schedule)
3. Lambda runs your code when triggered
4. Auto-scales from 1 to thousands of concurrent runs
Key Constraints
Max execution time: 15 minutes. If your function can't complete in 15 min, split it up or use EC2/Fargate instead. Supports Python, Java, Node.js, C#, Go, and custom runtimes.
Pricing
Pay per request + per GB-second of execution time. 1 million requests/month free. No charge when function isn't running — ideal for spiky or infrequent workloads.
Real-World Use Cases
Resize images on S3 upload, process DynamoDB stream changes, run scheduled cron jobs, respond to API Gateway calls, send emails when a form is submitted.

Containers on AWS — ECS, EKS, ECR, Fargate

Containers package your app + all dependencies into a portable unit. "Works on my machine" becomes "works everywhere." AWS offers an ecosystem for running containers at scale.
Amazon ECR
Container Registry — where you store and manage container images. Like Docker Hub, but inside AWS. Pull images from here to deploy.
Store Images
Amazon ECS
Elastic Container Service — AWS-native orchestration. Simpler, tightly integrated with AWS. Good for teams that want managed orchestration without Kubernetes complexity.
Orchestrate
Amazon EKS
Elastic Kubernetes Service — managed Kubernetes on AWS. Best for teams already using Kubernetes or needing fine-grained control. Open-source community support.
Kubernetes
AWS Fargate
Serverless compute for containers — works with both ECS and EKS. You don't manage any EC2 instances. Define CPU/memory, deploy containers, pay per use.
Serverless Hosting
Decision flow: Store images in ECR → orchestrate with ECS (simpler) or EKS (Kubernetes) → run on EC2 (more control) or Fargate (serverless).

Additional Compute Services — Purpose-Built Solutions

AWS Elastic Beanstalk
Deploy web apps by just uploading code. Beanstalk auto-provisions EC2, load balancers, scaling, and monitoring. You keep full visibility and control of the underlying resources.
Web Apps
AWS Batch
Run large-scale batch computing jobs (simulations, data processing, ML training). Auto-scales compute across EC2 fleet. No need to manage queuing or infrastructure.
Batch Jobs
Amazon Lightsail
Simplified VPS + storage + database at a predictable monthly price. Ideal for small businesses, blogs, simple apps, and learning cloud without AWS complexity.
Simple Hosting
AWS Outposts
Extends AWS infrastructure to your on-premises data center. Run AWS services locally for ultra-low latency, data residency requirements, or hybrid deployments.
Hybrid Cloud

Module 4 · Global Infrastructure

Regions, AZs, Edge Locations & CloudFormation

AWS operates a global network of data centers organized into Regions, Availability Zones, and Edge Locations — each serving a different purpose in building resilient, fast, globally distributed applications.

The Three Layers of AWS Infrastructure

AWS Regions
Geographic areas (e.g., US East, EU West, AP Southeast). Each Region is completely isolated — data never leaves without your permission. Every Region has ≥3 Availability Zones. Choose based on compliance, latency, features, and pricing.
Geographic Areas
Availability Zones (AZs)
Distinct data centers within a Region, each with independent power, networking, and cooling. Deploy across ≥2 AZs for high availability — if one AZ fails, others continue serving traffic seamlessly.
Fault Isolation
Edge Locations
Mini-data centers in cities worldwide (hundreds globally). Cache frequently accessed content close to users — images, videos, web assets. Part of the CloudFront CDN network. Separate from Regions/AZs.
Content Delivery

How to Choose a Region — The 4 Factors

FactorExplanationExample
1. ComplianceMost important. Some laws require data to stay within a country/region (GDPR, data sovereignty).EU customer data must stay in EU Regions
2. ProximityCloser to users = lower latency. Deploy in the Region nearest your customer base.Singapore users → Singapore Region
3. Feature AvailabilityNot all services are in all Regions. New features roll out progressively to Regions over time.Newer AI services may only be in us-east-1
4. PricingCosts vary by Region due to local taxes, energy costs, and infrastructure investment.Brazil Region is typically more expensive

Infrastructure as Code — AWS CloudFormation

CloudFormation lets you define your entire AWS infrastructure in a text file (JSON or YAML called a template). Instead of clicking through the console for every resource, you describe what you want and CloudFormation builds it. Deploy the same template across 10 different accounts or Regions and get identical environments every time.
Why IaC Matters
Manual provisioning is slow and error-prone. IaC is repeatable, version-controlled (track changes in Git), and eliminates "works in dev, broken in prod" environment drift.
CloudFormation Workflow
Write template → CloudFormation calls all needed AWS APIs → Resources created as a "Stack". Update the template, CloudFormation figures out what changed and updates just those resources.
Interacting with AWS
Console: GUI, best for beginners & visualizations. CLI: Command line scripts & automation. SDK: Code integration (Python, Java, etc.). CloudFormation: Infrastructure as code.

Module 5 · Networking

VPC, Subnets, Security Groups & Global Networking

Your private slice of the AWS cloud. Learn how to isolate resources, control traffic flow, connect on-premises networks, and deliver content globally with low latency.

Amazon VPC — Your Private Network in AWS

A Virtual Private Cloud (VPC) is a logically isolated section of the AWS cloud where you launch resources in a virtual network you define. Think of it as your own building inside AWS — you control what goes in, what goes out, and who can talk to what.
Public Subnets
Resources accessible from the internet. Connected to an Internet Gateway. Used for web servers, load balancers, NAT gateways. Drawn with dashed borders in diagrams.
Internet-Facing
Private Subnets
Resources isolated from the public internet. No direct internet access. Used for databases, app servers, internal services. Drawn with solid borders in diagrams.
Internal Only
Internet Gateway
The "front door" to your VPC for public internet traffic. Attach one to a VPC to allow inbound/outbound internet access to resources in public subnets.
Public Entry Point
Virtual Private Gateway
Connects your VPC to an on-premises network via VPN. Encrypted tunnel over the public internet. Only allows traffic from approved private networks.
VPN Entry Point

Network Security — Two Layers That Work Together

FeatureSecurity GroupsNetwork ACLs
Where?At the instance level (EC2, RDS)At the subnet boundary
StateStateful — remembers connections. Allow inbound, reply automatically allowed out.Stateless — checks every packet both ways. Must explicitly allow return traffic.
RulesAllow rules only. Default: deny all inbound, allow all outbound.Allow AND deny rules. Default: allow all (built-in) or deny all (custom).
AnalogyDoorman at your apartment building — remembers you came in, lets you back out.Passport control — checks your papers EVERY time you enter or exit, no memory.

Connecting to the AWS Cloud — 4 Options

AWS Client VPN
Connect individual remote workers to AWS or on-premises. Fully managed, elastic VPN that scales automatically. No hardware to manage. Best for remote employees.
Remote Workers
AWS Site-to-Site VPN
Connect an entire office or data center to AWS via encrypted VPN over the internet. High availability. Best for branch offices connecting to the cloud.
Office to Cloud
AWS PrivateLink
Privately connect your VPC to other VPCs, AWS services, or third-party services as if they were in your own network. No internet gateway needed, no public IPs.
Private VPC-to-VPC
AWS Direct Connect
Dedicated physical fiber connection from your data center to AWS. Bypasses the internet entirely. Lower latency, consistent bandwidth, better for large data transfers and compliance.
Dedicated Line
VPN vs Direct Connect:
VPN = encrypted tunnel over shared internet → flexible, lower cost, but variable performance.
Direct Connect = your own private physical wire → consistent performance, higher cost, better for heavy workloads and compliance.

Global Networking — DNS, CDN & Performance

Amazon Route 53
AWS's DNS service — translates human-readable domain names (anycompany.com) into IP addresses (192.0.2.0). Also supports domain registration, health checks, and routing policies (latency, geolocation, failover, weighted).
DNS + Domain
Amazon CloudFront
Content Delivery Network (CDN). Caches copies of content at edge locations worldwide. Users get content from the nearest edge location — faster load times, lower origin server load. Works with S3, EC2, API Gateway.
CDN
AWS Global Accelerator
Routes application traffic through the AWS private global network (not the internet). Provides two static IPs globally. Improves performance and availability for latency-sensitive apps like gaming or financial services.
Network Acceleration
CloudFront vs Global Accelerator:
CloudFront = caches content at the edge (images, videos, static files). Best for web content delivery.
Global Accelerator = routes traffic through AWS backbone to your app servers. Best for dynamic content, APIs, gaming.

Module 6 · Storage

S3, EBS, EFS, FSx & Storage Gateway

Three storage types, dozens of services. Block storage for databases. Object storage for files. File storage for shared access. Understand which to use when.

Three Types of Cloud Storage

TypeHow it worksAWS ServiceBest For
Block StorageData split into fixed blocks like a hard drive. Update just the changed block — efficient for frequent modifications.EC2 Instance Store, EBSDatabases, OS volumes, apps needing fast read/write
Object StorageFiles stored as self-contained objects with metadata + unique ID. Must rewrite entire object to update.Amazon S3Static files, backups, images, videos, data lakes
File StorageHierarchical file system shared across multiple instances simultaneously via NFS/SMB protocol.EFS, FSxShared workspaces, CMS, collaborative apps

Amazon S3 — Object Storage at Any Scale

S3 stores files as objects in buckets. Each object = data + metadata + unique key. Buckets are globally unique, you choose the Region. Max object size: 5TB. Bucket size: unlimited. Default: everything is private.
S3 Storage Classes — Access Frequency vs Cost
S3 Standard
Frequently accessed data. 3+ AZ replication. Used by default. Highest cost per GB stored.
Hot Data
S3 Intelligent-Tiering
Unknown access patterns. Auto-moves objects between tiers based on usage. Small monitoring fee but no retrieval fees. Best "set it and forget it" option.
Unknown Patterns
S3 Standard-IA
Infrequent access but instant retrieval needed. Lower storage cost, retrieval fee applies. Good for backups you rarely need.
Infrequent + Instant
S3 Glacier Instant
Archive with millisecond retrieval. Same speed as Standard but 68% cheaper. Medical images, news archives accessed occasionally.
Archive + Fast
S3 Glacier Flexible
Archive accessed 1-2x/year. Retrieval: 1-5 minutes (expedited) or 5-12 hours (bulk). Great for backups and disaster recovery.
Archive + Minutes
S3 Glacier Deep Archive
Lowest cost storage on AWS. Retrieval: 12 hours. For data kept 7-10 years for compliance (financial, healthcare, public sector).
Long-Term Archive
S3 Lifecycle Policies automate moving objects between storage classes as they age — e.g., move to Standard-IA after 30 days, Glacier after 90 days, delete after 365 days. Set it once, save money automatically.

Block Storage: Instance Store vs EBS

EC2 Instance StoreAmazon EBS
Persistence❌ Data LOST when instance stops/terminates✅ Data persists independently of instance lifecycle
AttachmentPhysically attached to the host machineNetwork-attached, can move between instances
PerformanceExtremely fast (NVMe SSD)Fast (provisioned IOPS available)
BackupNo built-in snapshotsEBS Snapshots to S3 (incremental, automated)
Use forTemp cache, buffers, scratch spaceDatabases, OS volumes, anything needing persistence

File Storage: EFS vs FSx

Amazon EFS
Fully managed Linux NFS file system. Scales automatically (petabytes). Multiple EC2 instances access simultaneously. Multi-AZ redundancy. Good for Linux workloads, shared dev environments, CMS.
Linux / NFS
FSx for Windows
Fully managed Windows file server with SMB protocol. Full Windows Active Directory integration. For Windows workloads needing NTFS, DFS, and Windows permission model.
Windows / SMB
FSx for Lustre
High-performance parallel file system. Sub-millisecond latency. Designed for HPC, ML training, video processing, financial simulations needing extreme throughput.
HPC / ML
AWS Storage Gateway
Bridge between on-premises storage and AWS cloud. 3 types: S3 File Gateway (files→S3), Volume Gateway (block→EBS snapshots), Tape Gateway (replace physical tapes with cloud). Enables gradual cloud adoption.
Hybrid Storage

Module 7 · Databases

RDS, Aurora, DynamoDB, ElastiCache & More

AWS philosophy: use the right database for the job. Relational for structured queries. NoSQL for flexible schemas and scale. In-memory for microsecond latency. Purpose-built for specific patterns.

Relational vs NoSQL — The Core Difference

Relational (SQL)NoSQL (Key-Value)
SchemaFixed, predefined. Every row has same columns.Flexible. Each item can have different attributes.
QuerySQL — complex joins, aggregations, relationshipsKey lookups, simple queries. Not designed for complex joins.
ScaleVertical scaling (bigger machine)Horizontal scaling (more machines)
Best forFinancial transactions, inventory, ERP, CRMUser profiles, gaming data, session data, catalogs
AWS ServiceRDS, AuroraDynamoDB

AWS Database Services Map

Amazon RDS
Managed relational database. AWS handles backups, patching, failover, and hardware. Supports MySQL, PostgreSQL, MariaDB, Oracle, SQL Server. Multi-AZ for high availability. Read replicas for scaling reads.
Relational
Amazon Aurora
AWS-built high-performance relational DB. MySQL/PostgreSQL compatible. 5x faster than MySQL, 3x faster than PostgreSQL. Auto-scales storage to 128TB. 6 copies across 3 AZs. Up to 15 read replicas.
High Performance SQL
Amazon DynamoDB
Fully managed serverless NoSQL. Key-value and document model. Single-digit millisecond latency at any scale. Auto-scales. No schema changes needed. Global tables for multi-region. Used by Amazon Prime Day (146M req/sec!).
NoSQL
Amazon ElastiCache
In-memory caching layer. Supports Redis, Valkey, and Memcached. Microsecond latency for frequently accessed data. Sits in front of RDS to reduce database load. Great for session data, leaderboards, real-time analytics.
In-Memory Cache
Amazon DocumentDB
MongoDB-compatible document database. For semi-structured JSON data that doesn't fit neatly in relational tables. Content management, catalogs, user profiles with flexible schemas.
Document / JSON
Amazon Neptune
Graph database. Designed for highly connected data — social networks, fraud detection, knowledge graphs, recommendation engines. Queries relationships that are impractical in relational DBs.
Graph
AWS Backup
Centralized backup management. Single dashboard to manage backups across EBS, EFS, RDS, DynamoDB, FSx, and on-premises. Automated schedules, retention policies, cross-region replication.
Backup Management
ElastiCache Pattern: App first checks cache → if found (cache hit) return immediately → if not (cache miss) query RDS, store result in cache, return. Subsequent requests for same data are microsecond-fast.

Module 9 · Security

IAM, Network Protection, Encryption & Detection

Security is a shared responsibility. AWS secures the infrastructure. You secure everything you put on it. The lifecycle: Prevent → Protect → Detect → Respond.

AWS Shared Responsibility Model

You (customer)AWS
Security IN the cloud: your data, apps, OS configuration, access management, network settingsSecurity OF the cloud: physical hardware, global network, hypervisor, data centers, Regions/AZs/Edge

IAM — Identity & Access Management

Default behavior: everything is denied. You must explicitly grant permissions. The principle of least privilege = give access only to what's needed, nothing more.
Root User
Account owner — unrestricted access. Enable MFA immediately. Never use for daily tasks. Create IAM users instead.
Account Owner
IAM Users
Represents a person or application. Has its own credentials. Zero permissions by default — must explicitly grant via policies.
Individual Identity
IAM Groups
Organize users with shared permissions. Assign a policy to a group, all members inherit it. Easier than managing permissions per-user.
Bulk Permissions
IAM Roles
Temporary identity — assume a role to get its permissions. Replaces all previous permissions while active. Used by EC2 instances, Lambda, cross-account access, federated users.
Temporary Access
IAM Policies
JSON documents that allow/deny specific actions on specific resources. Attach to users, groups, or roles. Fine-grained control: allow S3 read on this bucket, deny delete on all others.
Permission Rules
IAM Identity Center
Single sign-on (SSO) across all AWS accounts and apps. Connect to existing identity provider (Active Directory, Okta). Federated identity management — one login, access everywhere.
SSO / Federation

Data Encryption

Encryption at Rest
Data stored in S3, EBS, DynamoDB is encrypted. S3 encrypts all objects by default. EBS volumes can be encrypted. DynamoDB uses KMS automatically.
Encryption in Transit
Data moving between systems protected by TLS/SSL. Managed by AWS Certificate Manager (ACM). The padlock icon in your browser = TLS certificate.
AWS KMS
Create and manage cryptographic keys. Keys never leave KMS. Use across S3, EBS, RDS, Lambda. Specify which IAM users can use each key. Disable keys without deleting them.
Amazon Macie
ML-powered service that discovers and classifies sensitive data in S3 — PII, financial data, healthcare records. Helps with compliance (GDPR, HIPAA).

Amazon CloudWatch — Monitor Everything

Metrics
Collect numerical data from all AWS resources (CPU%, network bytes, request count). Create custom metrics for your application. Data retained for up to 15 months.
Alarms
Trigger actions when metrics cross thresholds. Send SNS notification, Auto Scale EC2 fleet, or invoke Lambda. "CPU > 80% for 5 minutes → add 2 instances."
Dashboards
Customizable real-time visualizations. Create a single "mission control" view of your entire infrastructure. Auto-refreshes. Shareable across teams.
Logs
Centralize and search log files from all AWS services and on-premises servers. Filter for error codes, search by time range, set alarms on log patterns.

AWS CloudTrail — Audit Every API Call

Every action in AWS is an API call. CloudTrail records every single one — who did it, what they did, when, from where, and whether it succeeded. Think of it as a comprehensive security camera for your entire AWS account.
CloudTrail Events
90 days of management event history viewable in console at no charge. Who launched that EC2 instance? Who deleted that S3 bucket? All answerable via CloudTrail.
CloudTrail Logs

AWS Pricing, Billing Tools & Support Plans

Three pricing principles, three cost drivers, five support tiers. Know how AWS charges you, which tools manage costs, and which support plan fits your needs.

3 Pricing Principles + 3 Cost Drivers

Pay As You Go
No upfront costs, no long-term commitments unless you choose. Only pay for resources while you use them. Stop the instance, stop the billing.
Save When You Commit
1 or 3 year commitment on compute usage → up to 66% off On-Demand. Savings Plans are flexible (any instance type). Reserved Instances lock in a specific type.
Pay Less by Using More
Volume discounts — the more you use, the less per unit. S3 pricing tiers decrease as GB stored increases. Data transfer rates drop at higher volumes.

Module 13 · Well-Architected Solutions

AWS Well-Architected Framework & Specialized Services

Building in the cloud isn't just about what works — it's about what works well. The 6 pillars of the Well-Architected Framework help you evaluate and continuously improve your architecture.

6 Pillars of the Well-Architected Framework

1. Operational Excellence
Run operations as code. Make frequent, small, reversible changes. Anticipate failure. Automate deployments via CI/CD pipelines. Continuously improve operations over time.
Automate Ops
2. Security
Apply security at every layer. Enable traceability. Automate security best practices. Protect data in transit and at rest. Keep people away from data. Prepare for security events.
Least Privilege
3. Reliability
Automatically recover from failure. Test recovery procedures. Scale horizontally. Stop guessing capacity. Manage change with automation. Multi-AZ and multi-region for resilience.
Survive Failures
4. Performance Efficiency
Use the right compute type. Rightsize instances. Democratize advanced technologies. Go global in minutes. Use serverless architectures. Experiment often.
Right-Size
5. Cost Optimization
Practice cloud financial management. Adopt consumption model (pay per use). Measure efficiency. Stop spending on undifferentiated heavy lifting. Analyze and attribute expenditures.
Reduce Waste
6. Sustainability
Understand your environmental impact. Establish sustainability goals. Maximize utilization. Anticipate and adopt more efficient hardware. Use managed services to reduce footprint.
Minimize Footprint
The AWS Well-Architected Tool is a free self-service tool. Create a workload, answer questions against your AWS architecture, and get a report showing risks and improvement recommendations aligned to each pillar.
